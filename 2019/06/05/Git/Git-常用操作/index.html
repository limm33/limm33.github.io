Git 常用操作

发表于 2019-06-05 | 分类于 Git

基本操作

  1. 初始化一个仓库：
    git init
  2. 添加到暂存区
    git add <文件名>
  3. 从暂存区移除
    git rm –cache 文件
  4. 提交到版本库
    git commit -m “chore: 提交理由”
  5. 查看仓库状态
    git status
  6. 查看提交历史
    git log

工作区&暂存区

  1. 工作区、版本库
  2. Git的版本库里存了很多东西，其中最重要的就是称为stage（或者叫index）的暂存区，还有Git为我们自动创建的第一个分支master，以及指向master的一个指针叫HEAD
  3. 第一步是用git add把文件添加进去，实际上就是把文件修改添加到暂存区；第二步是用git commit提交更改，实际上就是把暂存区的所有内容提交到当前分支。

版本回退

  1. git中用HEAD表示当前版本，HEAD^表示上一个版本，HEAD^^表示上上个版本，往上100个版本就是HEAD～100
  2. 回退：
    git reset –hard HEAD^ 回退到上一个版本

修改

git管理的是修改，而不是文件，比如：
第一次修改 ，git add ，再次修改，git commit ，第二次“再次修改”是无法被提交的。

撤销修改

  1. 如果只修改了，但是没有执行git add,执行git checkout – 可以撤销修改
  2. 如果执行了git add ,可以git reset HEAD 撤销git add 操作，重复1步骤就可以撤销修改了
  3. 如果commit了，可以进行版本回退 git reset –hard HEAD~2

删除文件

  1. 删除文件
  2. git add/rm
  3. git commit

远程仓库

添加远程库

  1. git remote add origin git@github.com:limm/test.git
  2. git push -u origin master
  3. 由于远程库是空的，我们第一次推送master分支时，加上了-u参数

从远程克隆

git clone 地址

分支管理

创建于合并分支

在版本回退里，每次提交，Git都把它们串成一条时间线，这条时间线就是一个分支。HEAD严格来说不是指向提交，而是指向master、其他，master才是指向提交的，所以，HEAD指向的就是当前分支。或者是当前分支的某个commitid

在dev中修改文件后，切换到master：
git merge dev
Updating d46f35e..b17d20e
Fast-forward
readme.txt | 1 +
1 file changed, 1 insertion(+)
Fast-forward代表“快进模式”，直接吧msater指向dev当前的提交，所以合并速度很快。
删除分支：
git branch -d dev
常用命令总结：

  1. 查看分支： git branch
  2. 创建分支： git branch
  3. 切换分支： git checkout
  4. 创建加切换分支 git checkout -b
  5. 合并某分支到当前分支： git merge
  6. 删除分支： git branch -d

    解决冲突

    当git merge后发生冲突的时候，执行git merge –abort可以撤销merge
    Git用<<<<<<<，=======，>>>>>>>标记出不同分支的内容。
    <<<<<<<HEAD 和=====之间的内容是我们本地的修改。

git log –graph 可以查看合并的图

分支管理策略

合并的时候Git会用到Fast forward模式，删除分支会丢掉分支信息，如果禁用Fastforwar模式，会merge时生成一个commit，从分支上就可以看出分支信息
git merge –no-ff -m “merge with no-ff” dev
因为本次合并要创建一个新的commit，所以加上-m参数，把commit描述写进去。
master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；

stash

当在dev分支上开发的时候突然后bug，我们执行了很多git add ，但是我们没有完成一个功能，不能commit的时候，我们可以使用：
git stash 会将文件放到“工作现场”保存起来，当我们切换到其他的分支解决完成所有的bug的时候，我们可以回到当前分支，执行：
git stash 可以查看工作现场
git stash apply 恢复所有的add提交
git stash drop 删除工作现场
或者直接使用git stash pop，恢复并删除工作现场
PS: 新增文件不是git管理范围，如果在一个分支上创建了新文件，并没有add，那切换分支的时候还会有这个文件

Feature分支

添加一个分支,添加一个新功能的时候，试验性的代码会把主分支搞乱，所以每添加一个新功能，最好新建一个feature分支，开发完成后合并，最后删除feature分支
git checkout -b <分支名称>

多人协作

当从远程仓库clone时，Git自动把本地的master分支和远程master 分支对应起来，并且远程仓库默认名称是origin
要查看远程库的信息，用git remote
git remote -v
推送分支：
git push origin <分支名称>

推送新建的分支到远程分支：
git push –set-upstream origin <远程分支名称>
删除远程分支命令
git push origin –delete <分支名称>

查看分支合并历史图：
git log –graph –pretty=oneline –abbrev-commit

更新代码：
git pull
git pull”操作等于“git fetch + git merge refs/remotes/origin/master“
git fetch 不会合并代码，git pull会合并代码

标签管理

发布一个版本时，我们通常先在版本库中打一个标签（tag），这样，就唯一确定了打标签时刻的版本。将来无论什么时候，取某个标签的版本，就是把那个打标签的时刻的历史版本取出来。所以，标签也是版本库的一个快照。
Git的标签虽然是版本库的快照，但其实它就是指向某个commit的指针（跟分支很像对不对？但是分支可以移动，标签不能移动），所以，创建和删除标签都是瞬间完成的。tag就是一个让人容易记住的有意义的名字，它跟某个commit绑在一起。

创建标签

git tag v1.0

git tag 查看所有的tag
git tag v0.9 f52c633 将标记打到某个commitid上
git show v0.9

操作标签

删除：git tag -d V0.1
创建的标签只存在本地，不会推送到远程，如果需要推送到远程：
git push origin
全部推送：
git push origin –tags

删除远程tag，先从本地删除，然后从远程删除
git tag -d
git push origin :refs/tags/v1.0

自定义Git

让Git显示颜色，会让命令输出看起来更醒目：
$ git config –global color.ui true

.gitignore

检查App.class在忽略文件中添加了：
git check-ignore -v App.class

别名

git config –global alias.st status （st将代替status）
